## 📋 Описание

Snake Quest - это современная интерпретация классической игры "Змейка" с множеством дополнительных функций, неоновым ретро-стилем и адаптивным дизайном.

### 🎮 Возможности

- **Различные карты**: Классическая, Лабиринт, Открытое поле, Зигзаг, Случайная генерация
- **4 уровня сложности**: Легкий, Средний, Сложный, Эксперт
- **Таблица лидеров**: Сохранение лучших результатов в localStorage
- **Неоновый ретро-стиль**: Анимированные эффекты, свечение, сетка
- **Адаптивный дизайн**: Поддержка мобильных устройств и разных разрешений
- **Полная типизация**: TypeScript с strict режимом
- **Поддержка русской раскладки**: Управление клавишами ЦФЫВ

## 🧠 Архитектурные решения и обоснование выбора библиотек

### Почему `useReducer` вместо Zustand или Redux?

В проекте используется **`useReducer`** вместо популярных стейт-менеджеров. Это решение принято на основе следующих соображений:

#### 1. **Отсутствие внешних зависимостей**
- `useReducer` является встроенным хуком React, не требует установки дополнительных пакетов
- Уменьшает размер финального бандла (нет необходимости включать Redux или Zustand)
- Снижает риск конфликтов версий и упрощает обновление зависимостей

#### 2. **Предсказуемость и простота отладки**
- Все изменения состояния происходят через четко определенные действия (actions)
- Код редьюсера легко читается и тестируется изолированно
- Встроенная поддержка TypeScript обеспечивает типобезопасность действий

#### 3. **Соответствие масштабу проекта**
- Игра не требует глобального состояния, разделенного между удаленными компонентами
- Вся логика состояния связана с одним корневым компонентом `App`
- `useReducer` идеально подходит для средних по сложности приложений (до 10-15 экранов)

#### 4. **Производительность**
- `useReducer` не создает подписчиков и не требует селекторов
- Обновления состояния происходят синхронно и предсказуемо
- Нет оверхеда, связанного с работой внешних библиотек

Почему нет React Router?
В проекте не используется React Router по следующим причинам:

1. Линейная структура навигации
Приложение имеет последовательную структуру переходов (меню → выбор карты → выбор сложности → игра)

Всего 5 экранов, которые легко переключаются через состояние status

React Router добавил бы избыточную сложность для такого простого сценария

2. Сохранение состояния между экранами
При переключении между экранами нужно сохранять состояние игры (имя игрока, выбранные карты)

С React Router пришлось бы использовать контекст или поднимать состояние выше

Текущая реализация с useReducer позволяет легко управлять переходами без дополнительных инструментов

3. Производительность и размер бандла
React Router добавляет ~10KB к бандлу

При текущей простоте приложения это неоправданно

Переключение через состояние происходит быстрее, чем через роутер
// Управление экранами через состояние
switch (state.status) {
  case 'menu': return <StartScreen />;
  case 'selectingMap': return <MapSelector />;
  case 'playing': return <GameCanvas />;
}

Почему CSS Modules?
Изоляция стилей: Стили компонентов не конфликтуют между собой

Предсказуемость: Классы генерируются уникальными именами

TypeScript поддержка: Автодополнение для CSS классов

Нет внешних зависимостей: Встроено в Vite

Почему Vite вместо Create React App?
Быстрая сборка: Vite использует esbuild, что значительно ускоряет разработку

Мгновенный HMR: Обновления происходят почти мгновенно

Меньше конфигурации: Настройка TypeScript, CSS Modules из коробки

Современные стандарты: Поддержка ES модулей "из коробки"
